1. QUÉ ES BRON
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Sistema POS/ERP propio para reemplazar Professional Plus en una ferretería /
materiales de construcción B2B en Argentina. El dueño es Rolo (único desarrollador).

Estrategia: correr en paralelo con Professional Plus un largo período de prueba
antes de pasar a producción. AFIP aún NO integrado (se agrega antes de producción).

Hardware: PCs viejas, Chromium en modo kiosk, servidor local XAMPP.

No se copian los PDFs de Professional Plus — el sistema nuevo genera sus propios
comprobantes.

Professional Plus (legado):

POST /api/cc                        ← { cliente_id, monto, tipo: pago|cargo }

GET  /api/filtros               ← { marcas[], proveedores[], categorias[] }

6. MIGRACIÓN DESDE PROFESSIONAL PLUS — ESTADO Y DECISIONES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Qué se migró:
  - Productos: 51,156+ (activos + borrados que aparecen en ventas 2026)
  - Clientes: solo los que tienen ventas en 2026
  - Ventas 2026: 12,472 (REMITOX: 5.942 / FC B-ELECT: 5.624 / FC A-ELECT: 867 / NC: 36 / FC B-FISCAL: 3)
  - Items de ventas: de stock_movimiento WHERE tipo=0, enlacepadreid=venta
  - Movimientos CC: de clientescc WHERE Fecha >= 46023 (= 2026-01-01)

Decisiones importantes:
  - Stock negativo → se pasa a 0 (GREATEST(s.Cantidad, 0))
  - Productos BORRADO=1 que aparecen en ventas 2026 → activo=0 (referencia histórica)
  - precio_unitario de venta_items: lookup en precios_movimientos (mejor aproximación, no exacto)
  - categoria: NULL en todos (familia_subfamilia_producto vacía en copia XAMPP)
  - proveedor: solo 3,041 de 51k productos tienen proveedor (normal, no todos están linkeados)
  - tipo_comprobante: viene de professionalplus.comprobantes.Nombre (JOIN por tipofactura)
  - Fecha Delphi: entero = días desde 1899-12-30 → DATE_ADD('1899-12-30', INTERVAL Fecha DAY)

Backup de Professional Plus:
  - Formato: .bak.g = gzip de mysqldump (magic bytes 1F 8B)
  - No fue necesario importarlo: la copia en XAMPP port 3307 ya tenía los datos

7. POS (pos/index.html) — FUNCIONALIDADES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Atajos de teclado:
  F2  → Búsqueda rápida de producto (dropdown inline, debounce 250ms)
  F3  → Modal avanzado de búsqueda de productos
  F4  → Búsqueda de cliente
  F10 → Confirmar venta

Modal F3 (búsqueda avanzada):
  - Filtros: Proveedor, Marca, Rubro (cargados de /api/filtros una vez)
  - Toggle "Con stock"
  - Switch Exacto / Similar
  - Tabla: Código | Nombre/Marca | Proveedor | Rubro | Stock | Precio
  - Navegación: ↑↓ Enter para seleccionar, Escape para cerrar
  - Stock coloreado: verde (>0), amarillo (bajo mínimo), rojo (<0)

Comprobantes disponibles: REMITO, FC B-ELECT, FC A-ELECT, PRESUPUESTO
Formas de pago: efectivo, transferencia, cc, tarjeta, cheque
  → Si pago=cc: registra en cuenta_corriente_movimientos y actualiza saldo cliente

Lógica al confirmar:
  1. INSERT ventas
  2. INSERT venta_items (por cada ítem)
  3. UPDATE productos stock_actual -= cantidad (por cada ítem)
  4. INSERT movimientos_stock tipo='venta' (por cada ítem)
  5. Si cc: INSERT cuenta_corriente_movimientos + UPDATE clientes.saldo_cc

8. MÓDULOS SECUNDARIOS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ventas.html — Historial de ventas
  Filtros: fecha desde/hasta, tipo comprobante, cliente (live search dropdown
           que filtra por cliente_id server-side), monto mín/máx, búsqueda libre
           (busca en número, nombre cliente Y monto vía CAST(FLOOR(total) AS CHAR))
  Agrupación client-side (sin re-fetch): sin agrupar / por cliente / por tipo
  Click en fila → modal con detalle completo del comprobante + items

cuentacorriente.html — Cuenta corriente
  Búsqueda live de cliente → card con nombre, CUIT, condición IVA, saldo, límite
  Tabla de movimientos con saldo acumulado por fila
  Botón "Registrar pago" → form inline → POST /api/cc

stock.html — Ajuste de stock
  Búsqueda live de producto → card con stock actual (coloreado), mínimo, precio, costo
  Tipos de ajuste: entrada (+cantidad), salida (-cantidad), ajuste (fijar valor absoluto)
  Historial de movimientos del producto seleccionado

compras.html — Ingreso de compras
  Layout similar al POS: busca productos → lista de ítems editables (cantidad + costo)
  Al confirmar: actualiza stock_actual Y costo_actual de cada producto
  Historial de últimas 10 compras visible a la derecha
