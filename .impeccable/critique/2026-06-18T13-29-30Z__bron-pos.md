## Anti-Patterns Verdict

**LLM assessment**: Not AI-generated. The interface passes the slop test cleanly. Argentine-specific vocabulary (remito, CUIT, Consumidor Final, cuenta corriente), hand-authored micro-interactions (cod-ok/cod-err barcode feedback, envio-reveal grid-rows transition, selec-fill keyframe), and genuinely different layouts across screens all mark this as intentional human work. The dark bordeaux primary + warm beige ground is unusual for generic SaaS. One candidate tell: the 11px/uppercase/letter-spacing:.5px label pattern appears verbatim ~20 times — intentional and consistent, not slop.

**Deterministic scan**: 8 findings across 5 files.
- **3 × dark-glow** (compras.html:74, index.html:341, ventas.html:70): `box-shadow: 0 2px 8px rgba(102,11,11,.3)` on primary buttons. **False positives** — these are subtle elevation shadows on bordeaux buttons over a light beige background (#F0EEEB). Not dark-mode glows.
- **2 × em-dash-overuse** (cuentacorriente.html, index.html): Em-dashes are in CSS section comment dividers (`/* ── LAYOUT ── */`), not body copy. **False positives**.
- **1 × side-tab** (ventas.html:214): `border-left:3px solid var(--azul)` on selected table rows. **Real finding** — redundant on top of the full-row background change already applied for selection. Remove; background-only is cleaner.
- **2 × flat-type-hierarchy** (productos.html, stock.html): 7 font sizes from 10–16px. **Real finding** — insufficient visual contrast between hierarchy levels. Collapse to 4 sizes.

**Visual overlays**: Browser visualization not attempted this run.

## Overall Impression

A solid, domain-specific POS built by someone who understands the business. The scanner workflow is genuinely good; the payment system handles real Argentine commerce edge cases. But there's a gap between the interface's physical context (counter cashier, 100+ sales/day, under time pressure) and some UX decisions. A confirmed sale that's wrong requires delete + full reentry. An AFIP-failing invoice type can be submitted without warning. The system offers no recovery anchor after confirmation. These aren't polish issues — they're daily friction points. The single biggest opportunity: wire the already-built edit-banner flow into Ventas so confirmed sales can have their line items corrected.

## What's Working

1. **Barcode scanner workflow**: The debounce-auto-add-clear-refocus cycle is purpose-built for a scanner environment. Green/red border flash gives feedback without requiring eyes off the counter. This is the right level of craft for the primary input method.

2. **Contextual action bars**: The sel-bar in POS and the seleccion-bar in Ventas appear on demand, echo selected context inline, and put operations where the user's attention already is. The selec-fill animation signals state change without blocking work.

3. **Payment edge cases handled correctly**: The pago-mixto system (5 methods, auto-complete, running "Restante" balance, confirm disabled until exact coverage) is correct for Argentine payment mix. The CC saldo shown at client selection prevents over-credit before commitment.

## Priority Issues

### [P0] No item-level edit on confirmed sales
**What**: "Modificar" in ventas.html only edits tipo_comprobante, tipo_pago, observaciones. Line items cannot be changed. Only path to correcting a wrong product/quantity: delete entire sale + reenter.
**Why it matters**: At 100–150 sales/day even 1% errors = 1–2 full corrections daily. 5–10 minutes each. The edit-banner flow (localStorage bron_editar_venta) was built for this purpose but isn't wired into the Ventas action bar.
**Fix**: Add "Editar ítems" to the seleccion-bar in ventas.html, loading the full sale into the POS via the existing editBanner flow. Infrastructure exists; needs a second entry point.
**Suggested command**: /impeccable harden pos/ventas.html

### [P0] No AFIP guard when selecting Factura Electrónica without valid fiscal data
**What**: Selecting FC A-ELECT or FC B-ELECT with Consumidor Final selected (or no client) produces no warning. The system will attempt AFIP submission with invalid fiscal data.
**Why it matters**: AFIP rejection creates accounting discrepancies, requires nota de crédito correction, exposes to audit risk. A new cashier has no way to know Factura A requires CUIT + Responsable Inscripto.
**Fix**: On tipo-comp change to FC A-ELECT/FC B-ELECT: validate client exists, has CUIT; for FC A validate condicion_iva = Responsable Inscripto. Show warning card above totals and disable btn-confirmar until resolved.
**Suggested command**: /impeccable harden pos/index.html

### [P1] Bulk adjustment toolbar too complex for a time-pressured context
**What**: Active sel-bar exposes 10 controls at 12px. Applying a 10% discount requires 8 steps.
**Why it matters**: Line-item discounts are common in B2B ferretería. Under counter pressure this becomes an error source, not a speed tool.
**Fix**: Replace ajuste section with a single "Aplicar ajuste" button opening a 3-field popover: tipo (toggle), value input, unit. Sel-bar active state: selected-count + Eliminar + Aplicar ajuste.
**Suggested command**: /impeccable distill pos/index.html

### [P1] No post-confirmation recovery anchor
**What**: After F10, a 3-second toast fires and screen resets to blank. Printer failure or customer question = navigate to ventas.html manually.
**Why it matters**: Printer failures are common in kiosk environments. The 3-second window is shorter than most printer response times.
**Fix**: After confirmation, show a persistent "Última venta" micro-panel at the base of the right column with sale number, total, Imprimir / Ver buttons. Disappears when first new item is added.
**Suggested command**: /impeccable harden pos/index.html

### [P2] Barcode scan failure gives no actionable feedback
**What**: Unknown code flashes red border for 700ms and resets. No message explains if product doesn't exist, is inactive, or scan was misread.
**Fix**: Show 2-second toast with failed code + search link. After 2 consecutive failures with same code, auto-open F3 modal pre-populated.
**Suggested command**: /impeccable harden pos/index.html

## Persona Red Flags

**Alex (Power User — keyboard-first daily cashier)**:
- No focus-return to barcode field after F4 client search. Must click or tab back.
- "Nueva venta" has no keyboard shortcut. 100+ mouse reaches per shift after each F10 confirmation.
- No keyboard navigation through items list for quantity editing. Mouse required for any quantity change.
- Ventas seleccion-bar actions (Ver/Comprobante/Modificar/Eliminar) are mouse-only; table rows have no keyboard focus/Enter handling.

**Jordan (Confused First-Timer — new staff member)**:
- Comprobante dropdown has no guidance. "Remito/Factura B/Factura A/Presupuesto" with no tooltip. Jordan may pick Factura A for Consumidor Final.
- 11px uppercase labels look like decorative UI, not active field labels. May not register Observaciones as interactive.
- sel-bar is not discoverable. Accidental item checkbox will partially light up the greyed bar with no explanation.

**Cajera de mostrador (Counter cashier under pressure)**:
- "Ítems" and "Unidades" rows add parsing steps before the TOTAL the cashier needs to read to the customer.
- After confirmation: no answer if customer asks "¿cuánto fue?" — the 3-second toast disappears.
- 8-step bulk discount process under counter pressure is unusable.

## Questions to Consider

1. Who owns the AFIP submission moment? If electronic invoice is processed at F10-confirmation time, the 3-second toast + immediate reset is wrong — the cashier needs the CAE before telling the customer the transaction is complete.

2. Is the bulk adjustment feature actually used daily? If not, a per-row discount popover directly on the item row would be faster and more discoverable.

3. What does credit limit enforcement look like in CC sales? The limit is displayed but not enforced. Is this by design (manager override) or an omission?
